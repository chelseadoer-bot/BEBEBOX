# 베베박스(BEBEBOX) 고객 여정 (Customer Journey)

> 육아 성장일기 + 베이비샤워 위시 + AI 놀이터(미니앱)를 하나의 **알(🥚) 포인트 경제**와
> **USERID(가족코드) 기반 데이터 파이프라인**으로 묶어, 키디키디(KidiKidi) 선물 트래픽으로
> 이어지는 바이럴 루프를 만든다.
>
> 이 문서는 (1) 고객/데이터 식별 체계, (2) 단계별 여정과 이벤트, (3) 포인트 경제,
> (4) 미니앱 통합, (5) 운영자 관점, (6) 데이터 무결성 점검 결과를 정리한다.

---

## 1. 고객 · 데이터 식별 체계 (USERID)

모든 데이터는 **가족코드(초대 코드, 대문자)** 하나를 USERID로 공유한다.

| 코드 소스 | 위치 | 값 |
|---|---|---|
| `getInviteCode()` | js/app.js | 초대 코드 |
| `getFamilyId()` | js/photo-storage.js | `getInviteCode()` 대문자 (fallback `BEBEBOX`) |
| 미니앱 `uid` | openAiApp → iframe `?uid=` | 초대 코드 대문자 |
| `norm_family()` | server.py | 입력값 대문자 정규화 |

→ **네 경로가 모두 동일한 값**으로 수렴한다. 즉 사진·가족상태·이벤트·미니앱 산출물이
전부 같은 USERID로 묶인다. (점검 결과: §6)

### 액터(actor)
- **parent** — 부모. USERID = 가족코드. 공간의 소유자.
- **family/spouse** — 배우자·조부모 등. 같은 초대 코드로 합류 → **같은 USERID**.
- **guest** — 지인. `/share/:코드` 방문자. 이름으로 식별, 이벤트 `user_id = "g:" + 이름`.

### 데이터 저장소 (모두 가족코드 키)
| 저장소 | 테이블/경로 | 내용 |
|---|---|---|
| 가족 상태 | `family_data` (JSON) | 프로필, 게시물, **알·쿠폰**, 위시리스트, 선물퍼즐, 공개여부, 선물한사람 |
| 사진 | `photos` | 업로드 이미지 |
| 여정 로그 | `events` | type, actor, name, item_id, meta, **user_id**, created_at |
| 미니앱 산출물 | `/data/ai/<slug>/app.db` | 미니앱별 records (uid=가족코드) |
| 로그인 | `sessions` | 카카오 세션 |

## 2. 단계별 여정 & 이벤트

```
가입 ─▶ 기록 ─▶ 미션/보상 ─▶ 공유 ─▶ 지인 유입(선물) ─▶ AI 놀이터 ─▶ 쿠폰교환 ─▶ 키디키디
```

### 2.1 가입 / 온보딩
카카오 로그인 → 부모/초대 선택 → (부모) 역할 → 아기 이름 → 키디키디 연동(선택) → 완료.
초대 코드 생성 = 이후 모든 데이터의 USERID.

### 2.2 기록 (Record)
사진 여러 장 + 글을 한 게시물로 작성.
- 이벤트 `record` · **+10알** · 오늘의 미션 퍼즐 **1조각**.

### 2.3 오늘의 미션 (달성조건 → 보상)
하루 9칸 퍼즐을 기록·공유·**미니앱 이용**으로 채우면 완성.
- 9칸 완성 시 **+100알 보너스**.

### 2.4 공유 (Share)
사진첩/공유 링크(`/share/:코드`)를 카카오 등으로 전파.
- 이벤트 `share` · **+30알**.

### 2.5 지인 유입 & 선물 (Guest → Gift)
지인이 공유 페이지 진입 후 반응·선물.
- `share_view`(진입) → `heart`/`comment`(반응) → `gift_click`(선물 시작) → `gift_done`(선물 완료 + 방명록).
- 선물 퍼즐(3×3)이 채워지며, **누가 무엇을 선물했는지**가 `giftedBy`로 적치.

### 2.6 AI 놀이터 (미니앱) — §4 상세
앱 진입 → 생성요청 → **결과 보기 결제 게이트** → 생성완료 → (선택)결과공유.

### 2.7 쿠폰 교환 (Exchange)
**1,000알 → 1,000원** 키디키디 장바구니 쿠폰.
- 이벤트 `coupon`. 쿠폰은 `family_data.coupons`에 적치.

## 3. 포인트(알 🥚) 경제

한 재화(알)로 **행동 → 보상 → 소비 → 교환** 루프를 닫는다. (js/points.js)

### 적립
| 행동 | 적립 | 비고 |
|---|---|---|
| 아이 기록 | +10 | `POINT_RULES.photo` |
| 사진첩/링크 공유 | +30 | `POINT_RULES.share` |
| 오늘의 미션 완성(9칸) | +100 | `POINT_RULES.missionBonus` |
| 미니앱 결과 공유 | +30 | `MINIAPP_RULES.shareReward` |

### 차감
| 행동 | 차감 | 비고 |
|---|---|---|
| 미니앱 결과 보기 | 0 / 10 / 20 / 30 | 앱별(§4) `MINIAPP_COST` |
| 쿠폰 교환 | -1,000 | `POINT_RULES.couponCost` |

알·쿠폰은 `family_data`로 서버에 동기화되어 재접속·재배포에도 보존된다.

## 4. 미니앱(AI 놀이터) 통합

### 4.1 결과 보기 결제 게이트 (중간 단계)
토큰 비용이 드는 앱은 **결과를 받기 전에 알을 차감하는 확인 단계**를 거친다.

```
[입력/생성 버튼]
   └▶ KD._emit('request')                  ← 행동 적치(track: miniapp_request)
   └▶ KD._gate()  ──postMessage──▶ 부모
        ├ 비용 0 앱        → 즉시 통과(모달 없음)
        └ 비용 > 0 앱      → "🥚 N알 쓰고 결과 보기" 모달
              ├ 확인 & 잔액충분 → 차감(miniapp_spend) → allow
              ├ 잔액부족        → "알이 부족해요" → deny
              └ 취소            → deny
   └▶ (allow) api/run 호출
        ├ 성공 → KD._emit('generated')      ← 서버가 고객 DB에 miniapp_generate 적치
        │                                      + 미션 1조각 + 동기화
        └ 실패 → KD._emit('failed')         ← 차감분 자동 환불(miniapp_refund)
   └▶ (선택) 결과 공유 → KD._emit('shared') ← +30알(miniapp_share)
```

핵심: **차감은 게이트에서만**, 생성 실패 시 **환불**, 캐시(동일입력) 결과는 추가 과금 없음.

### 4.2 앱별 결과 보기 비용 (`MINIAPP_COST`)
| 앱 | slug | 비용(알) | 근거 |
|---|---|---|---|
| 글로벌 작명소 | naming | **0** | 저토큰 텍스트 |
| 집안일 당번 | chores | **0** | 무-LLM(클라 게임) |
| 성향·기질 검사 | temperament | **0** | 저토큰 텍스트 |
| 아이 건강 체크 | health | 10 | 텍스트 |
| 아이와 나의 전생 | pastlife | 10 | 텍스트 |
| 낙서 심리 분석 | doodle | 20 | 이미지 입력(비전) |
| 브이로그 제작소 | vlog | 20 | 영상 합성 |
| AI 컨셉스튜디오 | studio | 30 | 이미지 생성(최고토큰) |

> 금액은 `js/app.js`의 `MINIAPP_COST` 한 곳에서 조정한다.

### 4.3 DB 연결 (미니앱 ↔ 고객 정보)
- 미니앱 산출물은 `/data/ai/<slug>/app.db`에 **uid=가족코드**로 적치된다.
- 생성 성공 시 서버(`ai_backend`)가 본체 고객 DB `events`에도 **같은 가족코드**로
  `miniapp_generate`를 남긴다 → 미니앱 활동이 고객 여정/운영자 대시보드에 통합 노출.

## 5. 운영자 관점 (`/admin`)

- **전체 KPI**: 가입 가족 수, 총 이벤트, 공유조회/선물클릭/선물완료/기록 등.
- **회원 검색 → 회원 상세**(USERID별):
  - 전환 퍼널: 👀조회 → 🛍선물클릭 → 🎁선물완료 · ❤️하트 · 💬댓글 · **🎮 미니앱**
  - **누구에게 줘야/받았는지**: 공개 위시 + 받은 선물(선물한 사람)
  - **누가 얼마나 조각을 썼는지**: 지인별 선물 조각 수·메시지
  - 최근 활동 타임라인(기록·공유·선물·미니앱 등 통합)

## 6. 데이터 무결성 점검 (USER 데이터 기반)

| 점검 항목 | 결과 |
|---|---|
| USERID 일관성 (부모) | ✅ `getInviteCode`=`getFamilyId`=미니앱 `uid`=`norm_family` 모두 동일(대문자 초대코드) |
| 미니앱 산출물 ↔ 고객 DB | ✅ 동일 가족코드로 `events.miniapp_generate` 적치 (member_detail에서 확인) |
| 게스트 식별 | ✅ `user_id="g:이름"`, 가족 데이터와 분리되어 덮어쓰기 방지(`state._synced`) |
| 포인트 영속성 | ✅ `family_data`로 동기화(재접속·재배포 보존) |
| 결제 안전성 | ✅ 게이트에서만 차감, 생성 실패 시 환불, 캐시 무과금 |
| 이벤트 → 퍼널 | ✅ 모든 type은 `counts`에 집계, 주요 단계는 funnel로 노출 |

### 이벤트 타입 레퍼런스
`record`, `share`, `share_view`, `heart`, `comment`, `gift_click`, `gift_done`,
`publish`, `coupon`, `ai_app`(앱 진입), `ai_concept`(컨셉 미리보기),
`miniapp_request`(생성요청), `miniapp_spend`(결제), `miniapp_refund`(환불),
`miniapp_generate`(생성완료·서버), `miniapp_share`(결과공유).
